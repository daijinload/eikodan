# lastshot 3スタック比較 — 「重い一覧画面」でのアプリ層差の検証

Rust / Node.js(Next.js) / PHP(Laravel) を **同じ DB・同じ画面・本番ビルド・同じ localhost** で
横並び計測した記録のうち、**「画面を大きくして同時アクセスを増やすと差がどう出るか」** を
調べたパート。カウンター（極小画面）の比較は [`lastshot-bench/`](./lastshot-bench/) にある。
ここはその続きで、**業務でありがちな重い一覧画面（受注明細レポート）** を題材にした。

> 計測日: 2026-06-14 / 実装は別ブランチ `feat/lastshot-3stack-compare`（`feature-report` ほか）。

## 何を知りたかったか

カウンター（数字＋ボタンだけ）で測ると、1リクエストの差は **0.05ms 対 4ms** とはいえ、
人間の知覚閾値（〜100ms）よりずっと下なので **1クリックでは誤差にしか感じない**。

そこで立てた仮説:

> **画面を大きくして（＝1リクエストの仕事量を増やして）、さらに同時アクセスを増やすと、
> アプリ層（言語ランタイム × アーキテクチャ）の差が体感・運用コストに効くレベルまで開くはず。**

これを確かめるため、**N 行の明細テーブルを 1 リクエストで SSR する重い画面**を 3 スタック
共通で作り、行数 `N` と同時接続数 `c` を振って測った。

## 実験設計（フェアにするための統制）

- **題材**: 受注明細レポート。`ID / 受注番号 / 顧客 / 状態 / 数量 / 単価 / 金額` の 7 列テーブル。
  ページングなしで全行を 1 画面に SSR する（業務の「明細グリッド／レポート出力」相当）。
- **データは 3 スタック同一 SQL**。マイグレーションせず Postgres の `generate_series(1, N)` で
  N 行を**合成**する。各列は計算で埋める（0 値が出ないよう値域を作ってある）。
  → **DB は 3 スタックとも完全に同じ仕事**をする。合計金額も 3 つとも `13,200,600`(N=1000) で一致。
  **だから観測される差は丸ごとアプリ層**（クエリ結果の取り回し＋オブジェクト生成＋テンプレ/SSR 描画＋HTTP）。
- **同じ画面**。3 スタックとも同一の HTML 構造・同一のインライン CSS。フレームワーク CSS の
  ビルド差を交絡させない。読み取り専用なので JS も無し（純粋なサーバ描画コストだけを比べる）。
- **本番相当ビルドで測る**:
  - lastshot … Rust release(stable) の axum、1 プロセスで全コア使用。
  - next … `next build` → `next start`（1 Node プロセス）。Server Component で SSR。
  - laravel … nginx + php-fpm（static 30 ワーカ）＋ opcache + preload + `artisan optimize`。
- 同じマシン・localhost・同じ瞬間に計測。負荷は oha。各行はウォームアップ（捨て）→ 本計測。

## 環境

Apple Silicon mac（darwin arm64・15 論理コア）/ ネイティブ PostgreSQL 17（unix ソケット）/
lastshot = Rust release(stable) axum・next = Next.js 15.5（`next start`）・
laravel = Laravel 13 on nginx + php-fpm 8.5（static 30 ワーカ）opcache+preload。
**oha（クライアント）と 3 サーバが同じ CPU を取り合う**点に注意（下記「落とし穴」）。

## 結果① 1リクエストの描画時間（c=1・無負荷）

行数を 200 → 1,000 → 5,000 と増やしたときの p50/p90/p99 と単一接続スループット。

### 200 行

| stack | p50(ms) | p90(ms) | p99(ms) | RPS |
|---|---|---|---|---|
| lastshot | **0.367** | 0.389 | 0.445 | 2695 |
| laravel  | 4.492 | 5.171 | 5.856 | 224 |
| next     | 8.669 | 12.011 | 14.868 | 109 |

### 1,000 行

| stack | p50(ms) | p90(ms) | p99(ms) | RPS |
|---|---|---|---|---|
| lastshot | **3.431** | 3.632 | 3.754 | 297 |
| laravel  | 6.348 | 7.523 | 8.316 | 158 |
| next     | 41.730 | 44.788 | 47.030 | 25 |

### 5,000 行

| stack | p50(ms) | p90(ms) | p99(ms) | RPS |
|---|---|---|---|---|
| lastshot | **9.210** | 9.839 | 10.537 | 112 |
| laravel  | 15.394 | 17.215 | 19.839 | 65 |
| next     | 207.766 | 217.414 | 226.643 | 5 |

5,000 行で **next は p50 が 208ms** ＝ 体感でモタつく領域。lastshot の約 22 倍、laravel の約 13 倍。

### 1行あたりのコスト分解（ざっくり線形）

p50 を「固定コスト ＋ 行数 × 単価」で 200〜5000 行に当てると、きれいに線形に乗る:

| stack | 固定コスト | 1,000 行あたりの追加 |
|---|---|---|
| lastshot | ほぼ 0 ms | **約 1.8 ms** |
| laravel  | 約 4 ms（毎リクエストの bootstrap） | **約 2.3 ms** |
| next     | ほぼ 0 ms | **約 41 ms**（React SSR） |

ポイントは **next の「1 行あたり約 41ms/1000 行」**＝ lastshot/laravel の **約 20 倍**。
React の `renderToString` がセル単位で仮想 DOM を組むため、**行数に比例して重くなる**。

## 結果② 同時アクセス（c=20・rows=1000）＝「投げまくる」

重い画面（1,000 行）に同時 20 本を当てたときの p50 とスループット上限:

| stack | p50(ms) | p90(ms) | p99(ms) | RPS（上限） |
|---|---|---|---|---|
| lastshot | **6.486** | 6.763 | 7.427 | **3075** |
| laravel  | 11.397 | 13.030 | 16.829 | 1720 |
| next     | 678.226 | 710.713 | 891.851 | 30 |

next は同時 20 本で **p50 が 0.68 秒・上限 30 RPS**。lastshot の約 **1/100**、laravel の約 **1/57**。
単一 Node プロセス上で CPU 律速の React SSR が**直列に詰まり**、待ち行列でレイテンシが秒に届く。
laravel は php-fpm 30 ワーカで**並列にさばける**ので、1 描画は lastshot より遅くても上限は崩れない。

## HTML サイズ（1,000 行）

| stack | HTML 本体 |
|---|---|
| laravel  | 322,291 bytes |
| lastshot | 323,283 bytes |
| next     | 772,779 bytes |

同じ見た目でも next は約 2.4 倍。RSC ペイロード（クライアントに渡す「描画済みツリーの再現データ」）が
HTML に上乗せされるため。転送量・パースコストもこのぶん効く。

## 結果③ ブラウザでの追試（実機・キャッシュオフ）

oha が測るのはサーバ処理時間。実際に**ブラウザ（キャッシュオフ）で `rows=5000` を開いた総時間**も
測ったところ、順序も桁もサーバ実測のまま再現した:

| stack | ブラウザ総時間 | （参考）サーバ実測 c=1 |
|---|---|---|
| lastshot | 約 21 ms | 9.2 ms |
| laravel  | 約 35 ms | 15.4 ms |
| next     | 約 280 ms | 207.8 ms |

ブラウザ総時間 ＝ サーバ描画 ＋ ネットワーク ＋ HTML パース ＋ 数千行の DOM 構築/描画。
上乗せ（十数 ms）は**どのスタックも同じ HTML なので同程度**で、順位・桁はサーバ実測どおり。
next だけは HTML が約 2.4 倍重いぶんパースも余計にかかる。**「サーバ描画が支配項」**という読みが、
合成負荷（oha）だけでなく実機ブラウザでも確認できた。

## 結論

- **小画面では誤差、大画面では誤差じゃない。** カウンターでは体感ゼロだった差が、5,000 行＋同時
  アクセスでは **p50 が 7ms 対 678ms（約 100 倍）** まで開く。仮説どおり「画面を大きくして投げまくると
  アプリ層差が支配的になる」。同じ DB・同じ SQL なのにこれだけ開くので、**DB はボトルネックではない。**
- **しかも順位が入れ替わる。** カウンターでは `next ≒ laravel`（どちらも lastshot に大差で負け）だったが、
  重い描画では **lastshot < laravel ≪ next**。
  - **laravel** の弱点は「毎リクエストの固定 bootstrap（〜4ms）」。大画面ではこれが薄まり、Blade の
    1 行描画は安いので**相対的にむしろ伸びる**。並列も php-fpm ワーカでさばける。
  - **next** の弱点は「React SSR が行数に比例して重い × `next start` が 1 プロセス」。大画面＋同時
    アクセスで**一番崩れる**。フロント主役のアーキテクチャが、サーバ描画の重い画面では裏目に出る。
  - **lastshot** は固定コストもほぼ 0・1 行も安い・全コア使用で、どの軸でも崩れない。
- **「言語の差」ではなく「言語 × アーキテクチャの差」。** 同じ Node でも、薄い JSON API
  （`/api/increment`）は速く、React SSR の重い画面は桁違いに遅い。何を測る画面かで結論が変わる。

## 実運用での示唆（Next.js が一番危ない、ただし常にではない）

- **「Next.js が常にダメ」ではなく「この画面（サーバで大きな表を描く）が React SSR の急所」。**
  薄い JSON API（`/api/increment`）なら Node でも速かった。**重いサーバ描画を React にやらせると裏目**で、
  業務の一覧／レポート画面はモロにそこを踏む ── というのが今回の刺さりどころ。
- **Next.js で踏むときの回避策**: 一覧をページング/仮想スクロールで分割して 1 リクエストの行数を
  減らす（描画コスト ≒ 行数 × 41ms/1000行 なので素直に効く）。または重い一覧だけ React SSR を通さない
  薄い経路で出す。同時アクセスの上限は別途プロセス多重化（PM2 等）が要る。
- **そもそも素で速い土台（lastshot）なら、その回避策が要らない。** これが今回いちばん効いている差。
  「速い基盤を選ぶ」と「遅い基盤を最適化で誤魔化す」は、かかる労力が違う。

- **小画面では誤差、大画面では誤差じゃない。** カウンターでは体感ゼロだった差が、5,000 行＋同時
  アクセスでは **p50 が 7ms 対 678ms（約 100 倍）** まで開く。仮説どおり「画面を大きくして投げまくると
  アプリ層差が支配的になる」。同じ DB・同じ SQL なのにこれだけ開くので、**DB はボトルネックではない。**
- **しかも順位が入れ替わる。** カウンターでは `next ≒ laravel`（どちらも lastshot に大差で負け）だったが、
  重い描画では **lastshot < laravel ≪ next**。
  - **laravel** の弱点は「毎リクエストの固定 bootstrap（〜4ms）」。大画面ではこれが薄まり、Blade の
    1 行描画は安いので**相対的にむしろ伸びる**。並列も php-fpm ワーカでさばける。
  - **next** の弱点は「React SSR が行数に比例して重い × `next start` が 1 プロセス」。大画面＋同時
    アクセスで**一番崩れる**。フロント主役のアーキテクチャが、サーバ描画の重い画面では裏目に出る。
  - **lastshot** は固定コストもほぼ 0・1 行も安い・全コア使用で、どの軸でも崩れない。
- **「言語の差」ではなく「言語 × アーキテクチャの差」。** 同じ Node でも、薄い JSON API

## 再現方法

3 スタックを本番起動してから（[`lastshot-bench/`](./lastshot-bench/) 参照）:

```sh
cd lastshot-bench
ROWS=1000 CONN=1  ./run report   # 1リクエストの描画時間(無負荷)
ROWS=5000 CONN=1  ./run report   # 行数を増やすほど差が開く
ROWS=1000 CONN=20 ./run report   # 同時アクセスでの p50 / スループット上限
```

ブラウザで直接触る場合（`?rows=N` で重さ可変・最大 100000）:

```
http://127.0.0.1:3003/report?rows=5000   # lastshot
http://127.0.0.1:3103/report?rows=5000   # next（ここだけ体感でモタつく）
http://127.0.0.1:3203/report?rows=5000   # laravel
```

## 落とし穴（数値の読み方）

- **localhost の CPU 取り合い**: oha と 3 サーバが同じマシンの CPU を共有する。特に c=20 の絶対値は
  他プロセスの負荷で上下する。**順序関係（lastshot < laravel ≪ next）は頑健**だが、絶対値は
  「同条件・同時刻での横並び」として読む。厳密には専用クライアントマシンから測るべき。
- **合成データ・読み取り専用**: 行は `generate_series` で作り、DB 書き込みは無い。実業務の一覧は
  JOIN や集計でもっと DB を使う ── ただしそれは 3 スタック共通なので**比較としては相殺**される
  （ここで見たいのは「DB が同じとき残りで何倍差がつくか」）。
- **serving モデルは各スタックの素のまま**（axum=全コア 1 プロセス / next=1 Node プロセス /
  laravel=php-fpm 30 ワーカ）。これも「言語 × アーキテクチャ」の一部として込みで見る。next を
  複数プロセス化（PM2 等）すれば同時アクセスの上限は上がるが、1 リクエストの描画コスト（41ms/1000行）は
  変わらない。
- **線形分解は 3 点フィットの目安**。固定コスト/単価はオーダー感をつかむためのもので、厳密値ではない。
