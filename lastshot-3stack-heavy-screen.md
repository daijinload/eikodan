# lastshot 3スタック比較 — 「重い一覧画面」でのアプリ層差の検証

Rust / Node.js(Next.js) / PHP(Laravel) を **同じ DB・同じ画面・本番ビルド・同じ localhost** で
横並び計測した記録のうち、**「画面を大きくして同時アクセスを増やすと差がどう出るか」** を
調べたパート。カウンター（極小画面）の比較は [`lastshot-bench/`](./lastshot-bench/) にある。
ここはその続きで、**業務でありがちな重い一覧画面（受注明細レポート）** を題材にした。

> 計測日: 2026-06-14 / 実装は別ブランチ `feat/lastshot-3stack-compare`（`feature-report` ほか）。

## 何を知りたかったか

カウンター（数字＋ボタンだけ）で測ると、1リクエストの差は **0.05ms 対 4ms** とはいえ、
人間の知覚閾値（〜100ms）よりずっと下なので **1クリックでは誤差にしか感じない**。

そこで立てた仮説:

> **画面を大きくして（＝1リクエストの仕事量を増やして）、さらに同時アクセスを増やすと、
> アプリ層（言語ランタイム × アーキテクチャ）の差が体感・運用コストに効くレベルまで開くはず。**

これを確かめるため、**N 行の明細テーブルを 1 リクエストで SSR する重い画面**を 3 スタック
共通で作り、行数 `N` と同時接続数 `c` を振って測った。

## 実験設計（フェアにするための統制）

- **題材**: 受注明細レポート。`ID / 受注番号 / 顧客 / 状態 / 数量 / 単価 / 金額` の 7 列テーブル。
  ページングなしで全行を 1 画面に SSR する（業務の「明細グリッド／レポート出力」相当）。
- **データは 3 スタック同一 SQL**。マイグレーションせず Postgres の `generate_series(1, N)` で
  N 行を**合成**する。各列は計算で埋める（0 値が出ないよう値域を作ってある）。
  → **DB は 3 スタックとも完全に同じ仕事**をする。合計金額も 3 つとも `13,200,600`(N=1000) で一致。
  **だから観測される差は丸ごとアプリ層**（クエリ結果の取り回し＋オブジェクト生成＋テンプレ/SSR 描画＋HTTP）。
- **同じ画面**。3 スタックとも同一の HTML 構造・同一のインライン CSS。フレームワーク CSS の
  ビルド差を交絡させない。読み取り専用なので JS も無し（純粋なサーバ描画コストだけを比べる）。
- **本番相当ビルドで測る**:
  - lastshot … Rust release(stable) の axum、1 プロセスで全コア使用。
  - next … `next build` → `next start`（1 Node プロセス）。Server Component で SSR。
  - laravel … nginx + php-fpm（static 30 ワーカ）＋ opcache + preload + `artisan optimize`。
- 同じマシン・localhost・同じ瞬間に計測。負荷は oha。各行はウォームアップ（捨て）→ 本計測。

## 環境

Apple Silicon mac（darwin arm64・15 論理コア）/ ネイティブ PostgreSQL 17（unix ソケット）/
lastshot = Rust release(stable) axum・next = Next.js 15.5（`next start`）・
laravel = Laravel 13 on nginx + php-fpm 8.5（static 30 ワーカ）opcache+preload。
**oha（クライアント）と 3 サーバが同じ CPU を取り合う**点に注意（下記「落とし穴」）。

## 結果① 1リクエストの描画時間（c=1・無負荷）

行数を 200 → 1,000 → 5,000 と増やしたときの p50/p90/p99 と単一接続スループット。

### 200 行

| stack | p50(ms) | p90(ms) | p99(ms) | RPS |
|---|---|---|---|---|
| lastshot | **0.367** | 0.389 | 0.445 | 2695 |
| laravel  | 4.492 | 5.171 | 5.856 | 224 |
| next     | 8.669 | 12.011 | 14.868 | 109 |

### 1,000 行

| stack | p50(ms) | p90(ms) | p99(ms) | RPS |
|---|---|---|---|---|
| lastshot | **3.431** | 3.632 | 3.754 | 297 |
| laravel  | 6.348 | 7.523 | 8.316 | 158 |
| next     | 41.730 | 44.788 | 47.030 | 25 |

### 5,000 行

| stack | p50(ms) | p90(ms) | p99(ms) | RPS |
|---|---|---|---|---|
| lastshot | **9.210** | 9.839 | 10.537 | 112 |
| laravel  | 15.394 | 17.215 | 19.839 | 65 |
| next     | 207.766 | 217.414 | 226.643 | 5 |

5,000 行で **next は p50 が 208ms** ＝ 体感でモタつく領域。lastshot の約 22 倍、laravel の約 13 倍。

### 1行あたりのコスト分解（ざっくり線形）

p50 を「固定コスト ＋ 行数 × 単価」で 200〜5000 行に当てると、きれいに線形に乗る:

| stack | 固定コスト | 1,000 行あたりの追加 |
|---|---|---|
| lastshot | ほぼ 0 ms | **約 1.8 ms** |
| laravel  | 約 4 ms（毎リクエストの bootstrap） | **約 2.3 ms** |
| next     | ほぼ 0 ms | **約 41 ms**（React SSR） |

ポイントは **next の「1 行あたり約 41ms/1000 行」**＝ lastshot/laravel の **約 20 倍**。
React の `renderToString` がセル単位で仮想 DOM を組むため、**行数に比例して重くなる**。

## 結果② 同時アクセス（c=20・rows=1000）＝「投げまくる」

重い画面（1,000 行）に同時 20 本を当てたときの p50 とスループット上限:

| stack | p50(ms) | p90(ms) | p99(ms) | RPS（上限） |
|---|---|---|---|---|
| lastshot | **6.486** | 6.763 | 7.427 | **3075** |
| laravel  | 11.397 | 13.030 | 16.829 | 1720 |
| next     | 678.226 | 710.713 | 891.851 | 30 |

next は同時 20 本で **p50 が 0.68 秒・上限 30 RPS**。lastshot の約 **1/100**、laravel の約 **1/57**。
単一 Node プロセス上で CPU 律速の React SSR が**直列に詰まり**、待ち行列でレイテンシが秒に届く。

> **ただしこの「約 1/100」は二つの別物が混ざった数字**なので、分けて読む必要がある。
>
> 1. **1 リクエストの描画コスト差（本質差・多重化しても消えない）** ── next は 1 描画が
>    41ms/1000行で lastshot/laravel の約 20 倍重い。これは **React SSR というアーキテクチャ由来**で、
>    プロセスを何個に増やそうが 1 リクエストあたりのコストは動かない。**これが本当に強い主張。**
> 2. **素のサービング構成での同時さばき差（構成差・設定を変えれば縮む）** ── 上限 RPS が 30 まで
>    落ちる主因は、**next が 1 Node プロセスで CPU バウンドな描画を直列にさばいている**こと。実際に
>    PM2 cluster で 15 プロセスに多重化すると（[`lastshot-bench/README.md`](./lastshot-bench/README.md)
>    の「追加検証」を参照）**RPS は 31 → 208（約 6.7 倍）・p50 は 646ms → 82ms** まで改善し、対 lastshot
>    の比率は **1/92 → 1/14** に縮む。これは言語の地力ではなく**起動構成の差**。
>
> 「100 倍」は (1)×(2) の積。**(1) は多重化で絶対に消えないが、(2) は (実測で) 約 1/7 まで小さくできる。**
> 「Next は同時アクセスで 100 倍遅い」と一語で受け取ると、強い主張 (1) が弱い主張 (2) の巻き添えで
> 過大評価される。なお実測の「1/14」は事前予測の「1/7」までは届いていない（同一 localhost で oha + 15
> next ワーカ + axum + Postgres + nginx + php-fpm が CPU を取り合うため）。**多重化しても逆転は起きない**
> ＝ React SSR の絶対コストが残るぶん、lastshot は依然 14 倍速い。

laravel が上限を崩さないのも半ば**構成の恩恵**で、lastshot＝1 プロセス 15 コアに対し laravel＝
**php-fpm 30 ワーカ**とプロセス数が倍違う。「並列にさばける」のは 30 ワーカという構成込みの話で、
純粋な言語の地力だけで崩れないわけではない（順位の解釈はこの構成差込みで読む）。

## HTML サイズ（1,000 行）

| stack | HTML 本体 |
|---|---|
| laravel  | 322,291 bytes |
| lastshot | 323,283 bytes |
| next     | 772,779 bytes |

同じ見た目でも next は約 2.4 倍。RSC ペイロード（クライアントに渡す「描画済みツリーの再現データ」）が
HTML に上乗せされるため。転送量・パースコストもこのぶん効く。

## 結果③ ブラウザでの追試（実機・キャッシュオフ）

oha が測るのはサーバ処理時間。実際に**ブラウザ（キャッシュオフ）で `rows=5000` を開いた総時間**も
測ったところ、順序も桁もサーバ実測のまま再現した:

| stack | ブラウザ総時間 | （参考）サーバ実測 c=1 | 上乗せ（＝ブラウザ−サーバ） |
|---|---|---|---|
| lastshot | 約 21 ms | 9.2 ms | 約 12 ms |
| laravel  | 約 35 ms | 15.4 ms | 約 20 ms |
| next     | 約 280 ms | 207.8 ms | **約 73 ms** |

ブラウザ総時間 ＝ サーバ描画 ＋ ネットワーク ＋ HTML パース ＋ 数千行の DOM 構築/描画。
**lastshot と laravel の上乗せは 12〜20ms で同程度**（同じ HTML なので当然）。だが **next だけは
上乗せも約 73ms と重い** ── HTML が約 2.4 倍（772KB）あるぶん、クライアント側のパースと DOM 構築も
余計にかかるからだ。つまり next は**サーバ描画が重いうえにクライアント上乗せも重い二重苦**で、
これは RSC ペイロードを HTML に積む設計コストが**実機でも**効いている証拠になる。
それでも**順位・桁はサーバ実測どおり**で、**「サーバ描画が支配項」**という読みは合成負荷（oha）
だけでなく実機ブラウザでも確認できた ── **合成ベンチが実体を取り逃していない**ことの裏取りでもある。

## 結論

- **小画面では誤差、大画面では誤差じゃない。** カウンターでは体感ゼロだった差が、重い描画（1,000 行）＋
  同時アクセス（c=20）では **p50 が 6.5ms 対 678ms（約 100 倍）** まで開く。仮説どおり「画面を大きく
  して投げまくるとアプリ層差が支配的になる」。同じ DB・同じ SQL なのにこれだけ開くので、**DB は
  ボトルネックではない。**（ただしこの 100 倍は「1 描画の重さ＝本質差」と「素のサービング構成＝構成差」の
  積で、後者はプロセス多重化で縮む ── 内訳は結果②の注記参照。）
- **しかも順位が入れ替わる。** カウンターでは `next ≒ laravel`（どちらも lastshot に大差で負け）だったが、
  重い描画では **lastshot < laravel ≪ next**。
  - **laravel** の弱点は「毎リクエストの固定 bootstrap（〜4ms）」。大画面ではこれが薄まり、Blade の
    1 行描画は安いので**相対的にむしろ伸びる**。並列も php-fpm ワーカでさばける。
  - **next** の弱点は二つ。**(本質)** React SSR が行数に比例して重い（41ms/1000行・多重化しても消えない）、
    **(構成)** `next start` が 1 プロセスで同時アクセスを直列にさばく（PM2 cluster × 15 で実測すると
    対 lastshot が 1/92 → 1/14 まで縮む）。大画面＋同時アクセスで両方が重なり**一番崩れる**が、
    消せるのは構成側だけ。フロント主役のアーキテクチャが、サーバ描画の重い画面では裏目に出る。
  - **lastshot** は固定コストもほぼ 0・1 行も安い・全コア使用で、どの軸でも崩れない。
- **「言語の差」ではなく「言語 × アーキテクチャの差」。** 同じ Node でも、薄い JSON API
  （`/api/increment`）は速く、React SSR の重い画面は桁違いに遅い。何を測る画面かで結論が変わる。

## 実運用での示唆（Next.js が一番危ない、ただし常にではない）

- **「Next.js が常にダメ」ではなく「この画面（サーバで大きな表を描く）が React SSR の急所」。**
  薄い JSON API（`/api/increment`）なら Node でも速かった。**重いサーバ描画を React にやらせると裏目**で、
  業務の一覧／レポート画面はモロにそこを踏む ── というのが今回の刺さりどころ。
- **Next.js で踏むときの回避策**: 一覧をページング/仮想スクロールで分割して 1 リクエストの行数を
  減らす（描画コスト ≒ 行数 × 41ms/1000行 なので素直に効く）。または重い一覧だけ React SSR を通さない
  薄い経路で出す。同時アクセスの上限は別途プロセス多重化（PM2 cluster × N など）が要る ── 実測で
  /report は RPS 31 → 208（約 6.7 倍）。Vercel/Cloud Run のような前段多重化も同じ役割。
- **そもそも素で速い土台（lastshot）なら、その回避策が要らない。** これが今回いちばん効いている差。
  「速い基盤を選ぶ」と「遅い基盤を最適化で誤魔化す」は、かかる労力が違う。

## 開発モード（dev）での速度 ── 「ローカルで普段感じる遅さ」の正体

ここまでは本番ビルドの話。だが**開発者がローカルで日常的に触るのは dev モード**で、
普段から体感している速度はこちらだ。そこで 3 スタックを開発モードで起動して同じ計測を当てた。

> 結論: **dev では next だけが激変**し、lastshot と laravel は本番とほぼ変わらない。
> 普段ローカルで感じる「もっさり」は、ほぼ **Next.js の dev モード特有の性質**だった。

### ウォーム時の dev vs 本番（c=1・p50）

**カウンター `GET /`**

| stack | dev | 本番 | 差 |
|---|---|---|---|
| lastshot | 0.068 ms | 0.053 ms | ほぼ同じ |
| next | **11.8 ms** | 0.76 ms | dev が約 15 倍遅い |
| laravel | 3.76 ms | 4.37 ms | ほぼ同じ |

**受注明細レポート `GET /report`（1,000 行）**

| stack | dev | 本番 | 差 |
|---|---|---|---|
| lastshot | 3.33 ms | 3.43 ms | ほぼ同じ |
| next | **320 ms** | 42 ms | dev が約 8 倍遅い |
| laravel | 6.48 ms | 6.35 ms | ほぼ同じ |

### 「編集してから初回アクセス」の冷えた状態（cold・初回 1 発の総時間）

dev で最も効くのが**ルートのオンデマンド再コンパイル**。ファイルを編集した直後に開く最初の 1 発:

| 初回ヒット | lastshot | next | laravel |
|---|---|---|---|
| `GET /` | 3.4 ms | **1,023 ms** | 93 ms |
| `GET /report`（1,000 行） | 4.5 ms | **667 ms** | 9 ms |

next は編集のたびに、そのルートを最初に開いた 1 発が**約 1 秒**かかる。

### なぜこうなるか

- **lastshot は dev＝本番速度。** Rust の debug ビルドはこのワークロードなら release とほぼ同速で、
  しかも**リクエスト毎のコンパイルが無い**。テンプレ変更はライブリロードで即反映され、それでも 0.068 ms。
  ただしこれは**今回が CPU 負荷の軽い描画系（DB 待ち＋文字列連結中心）だから**で、debug の「最適化なし」
  ペナルティ（数値計算やパースの重い処理なら 10〜50 倍にもなる）が表に出ないだけ。**「Rust は常に
  dev＝本番」ではなく、CPU を回す処理では debug もちゃんと遅くなる**点は注意。
- **laravel も dev ≈ 本番。** `artisan serve` は opcache 無しだが、ここの主コストは Blade 描画＋DB で
  本番と同じ。opcache が効くのはフレームワーク bootstrap（小さい）だけなので体感差が出ない。
  ただし `serve` は**単一ワーカ**なので、同時アクセスの上限は本番（php-fpm 30 ワーカ）より低い。
- **next だけ dev が別物。** dev は「未最適化の React ＋ オンデマンドコンパイル（Turbopack）＋
  ソースマップ＋HMR」。**編集するたびにルートが初回アクセスで再コンパイル（〜1 秒）**し、ウォームでも
  本番ビルドの**約 8 倍**遅い（1,000 行で 320 ms）。

つまり**本番ベンチ（next 42ms）より dev（320ms ＋ 初回 1 秒）のほうが体感はずっと悪い**。
ローカルで日々感じる差は、本番比較で見た差よりさらに大きい ── というのが dev モードでの答え。
（計測は各スタックを開発起動 ── lastshot＝debug ビルド / next＝`next dev` / laravel＝`artisan serve` ──
してウォーム時は oha、cold は編集直後の初回 1 発の総時間。）

### dev・ブラウザ実機での内訳 ──「lastshot が 185ms?」の正体

dev で `rows=5000` をブラウザ（キャッシュオフ・ウォーム）で開くと、総時間は一見
**lastshot 205 ms ≒ laravel 187 ms** で「lastshot のほうが速いはずでは?」と引っかかる。
だが総時間を **サーバ(TTFB) / 転送 / DOM構築** に分解すると、その引っかかりの正体が見える:

| stack | ブラウザ総計 | サーバ(TTFB) | 転送 | DOM構築 |
|---|---|---|---|---|
| lastshot | 205 ms | **15 ms** | ~1 ms | 139 ms |
| next | 2,250 ms | **1,754 ms** | 345 ms | 150 ms |
| laravel | 187 ms | **23 ms** | ~1 ms | 162 ms |

- **lastshot のサーバ処理は 15 ms**（curl 実測 17 ms と一致）で 3 スタック最速。総計 205 ms の大半は
  **DOM構築 139 ms ＝ ブラウザが 5,000 行を組む全スタック共通の作業**（laravel も DOM 162 ms）。
  → 総計だけ見ると lastshot ≒ laravel に見えるが、**サーバの地力は lastshot が上（15 vs 23 ms）**。
  「lastshot はもっと速いはず」という体感は正しく、**その速さが共通の DOM 構築に埋もれていた**だけ。
  つまり「185 ms」は lastshot が遅いのではなく、**ブラウザ側の行描画コストまで足し込んだ総時間**であって、
  サーバ処理だけを見れば 15 ms ── という**計測の切り分けミス**だった。
- **next の 2,250 ms はほぼ全部サーバの React SSR（1,754 ms）。** さらに dev の HTML は decoded
  7.3 MB（他スタックは 1.6 MB）で**転送 345 ms も重い**。しかもこれは**ウォーム（2 回目以降）の値**で、
  「初回コンパイルだけ遅い」のではなく**毎回およそ 2 秒**かかる。

## 再現方法

3 スタックを本番起動してから（[`lastshot-bench/`](./lastshot-bench/) 参照）:

```sh
cd lastshot-bench
ROWS=1000 CONN=1  ./run report   # 1リクエストの描画時間(無負荷)
ROWS=5000 CONN=1  ./run report   # 行数を増やすほど差が開く
ROWS=1000 CONN=20 ./run report   # 同時アクセスでの p50 / スループット上限
```

ブラウザで直接触る場合（`?rows=N` で重さ可変・最大 100000）:

```
http://127.0.0.1:3003/report?rows=5000   # lastshot
http://127.0.0.1:3103/report?rows=5000   # next（ここだけ体感でモタつく）
http://127.0.0.1:3203/report?rows=5000   # laravel
```

## 落とし穴（数値の読み方）

- **localhost の CPU 取り合い**: oha と 3 サーバが同じマシンの CPU を共有する。特に c=20 の絶対値は
  他プロセスの負荷で上下する。**順序関係（lastshot < laravel ≪ next）は頑健**だが、絶対値は
  「同条件・同時刻での横並び」として読む。厳密には専用クライアントマシンから測るべき。
- **合成データ・読み取り専用**: 行は `generate_series` で作り、DB 書き込みは無い。実業務の一覧は
  JOIN や集計でもっと DB を使う ── ただしそれは 3 スタック共通なので**比較としては相殺**される
  （ここで見たいのは「DB が同じとき残りで何倍差がつくか」）。
- **serving モデルは各スタックの素のまま**（axum=全コア 1 プロセス / next=1 Node プロセス /
  laravel=php-fpm 30 ワーカ）。これも「言語 × アーキテクチャ」の一部として込みで見る。next を
  複数プロセス化（PM2 cluster × 15 で実測: RPS 31 → 208・6.7 倍、対 lastshot 比は 1/92 → 1/14）すれば
  同時アクセスの上限は上がるが、1 リクエストの描画コスト（41ms/1000行）は変わらない。
- **線形分解は 3 点フィットの目安**。固定コスト/単価はオーダー感をつかむためのもので、厳密値ではない。
