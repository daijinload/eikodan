# 調査: 「DBが律速だから Node も Rust も速度は変わらない」は本当か

計測日: 2026-06-14 / 実装ブランチ: [`bench-rust-vs-node`](https://github.com/daijinload/eikodan/tree/bench-rust-vs-node)

> このドキュメントは調査結果のまとめです。**ベンチのコード一式・生成レポートは上記ブランチ**にあります
> （`lastshot/crates/feature-bench`, `lastshot-node/`, `lastshot-node/bench/REPORT*.md`）。
> 本 PR はドキュメントのみで、実装はマージしません。

---

## 一行結論

**「DBが遅いから Node も Rust も速度は変わらない」は、実測ではほぼ全領域で崩れる（＝速度は変わる）。
そして適切なドライバ（パイプライン）を使えば Rust が一貫して速い。**

- Rust が明確に勝つ: **1コア比較の点SELECT(2.4〜5.3×)・/ping(約2×)・重DBの CPU 効率(約3×)とテール安定**。
- 当初 Node が勝った **多コア × 点SELECT** も、Rust をパイプライン化(tokio-postgres)したら
  **~135k rps で Node(~121k)を逆転**。→ これは「言語」でなく「**ドライバがパイプラインするか**」の差だった。
- ほぼ同じになるのは **重DBクエリのスループットだけ**（それでも Rust は CPU 約1/3・テール安定）。

つまり同僚の主張は「最重量クエリのスループット」という1点だけを一般化したもので、軸を振ると崩れる。
ただし「Rust なら何でも速い」も誇張で、スタック/ドライバ/コア配分で勝者は変わる。

## 背景（なぜ測ったか）

「DB が律速だから言語/ランタイムは関係ない」という主張を、印象論でなく実測で検証する。
仮説は「主張が本当に成り立つのは **DB が CPU を使い切って律速になる狭い領域だけ**」で、
実 API の大半が住む「DB は軽い点取得」領域では言語/ランタイムが効く、というもの。
これを **DB の重さを軸にスライド**させて定量化した。

さらに、仮にスループットが収束する重 DB 領域でも、**Rust は同じ負荷を圧倒的に少ない CPU で・
テール(p99)安定で捌く**（＝「同じ速度」でも「同じコスト・同じ安定性」ではない）ことを示し、
「同じ速度」という結論の射程を限定する。

## 計測方法と公平性コントロール

### 測定環境（後で読む人/AI が誤読しないための条件一覧）

| 項目 | 値 |
| --- | --- |
| マシン / CPU | MacBook（macOS）・**Apple M5 Pro**・**15 コア（高性能 P×5 + 高効率 E×10）**・メモリ 48GB |
| Postgres の場所 | **ローカル同一ホスト**（homebrew native `postgresql@17`）。**unix ソケット `/tmp`**（TCP ではない／`DATABASE_URL` 未設定）。負荷生成・PG・被験サーバが**全て同一マシン同居**。 |
| 計測ツール | **oha 1.14.0**（自前スクリプトではない）。`--output-format json` で機械可読出力を回収。 |
| ウォームアップ | **あり**。各セルで本計測前に oha を **3 秒**流して捨て（`WARMUP=3`）、その後 **10 秒**本計測（`DURATION=10`）。加えて Node は JIT・pool 充填・PG プランキャッシュも温まる。 |
| 点SELECT(`/db/light`) のペイロード/行数 | `select value from counter where id = 1` = **主キー1点取得・返却1行**。レスポンスは `{"value":0}` ≈ **11〜13 バイト**。テーブル走査なし。 |
| 重クエリ(`/db/heavy`) | `bench_rows`(**30 万行**) を `like '%abc%'` で**全走査集約**（≈26ms の PG CPU）。＝DB 律速領域の対照。 |
| コネクションプール数 | `POOL_MAX`。**正典=16 / 追試=64**。全コアモードは各 worker `POOL_MAX/W` にして**総和を一致**。パイプライン版は別途 tokio-postgres を `PIPE_CLIENTS`(既定 `POOL_MAX`)本。 |
| 同時接続(oha `-c`) | 1 / 8 / 32 / 64 / 128 / 256 をスイープ。 |

**「1コア対1コア」をどう制約したか（幅の解釈に直結するので明記）**:
**taskset も cgroup も使っていない**（macOS に taskset は無く、コア cgroup 隔離もしていない）。
やったのは **アプリの並列度を 1 に揃えただけ** ── Rust = `TOKIO_WORKER_THREADS=1`、Node = cluster 無しの単一プロセス。
**コアのピン留めはしておらず、OS スケジューラ任せ**で、さらに oha と PG が同一マシンの他コアを使う。
したがってこの数字は「**ハード隔離した 1 物理コア同士**」ではなく「**等しい並列度予算（1 ワーカー）同士の
ランタイム効率比較**」と読むべき。点SELECT の **2.4〜5.3×** という幅は、この条件下（同時数 32→256 で拡大）の値である。
さらに M5 Pro は **P コア×5 + E コア×10 の非対称構成**なので、ピン留めしていない以上、Rust の 1 ワーカーと
Node の 1 プロセスが**別種のコア（P と E）に載った可能性**を排除できない。その場合、倍率の一部はコア種の差に由来しうる
── これは「並列度を 1 に揃える」だけでは潰しきれない測定限界として、正直に残しておく。
（全コアモードは 8 worker/8 thread。15 コア機なので残り約 7 コアを oha+PG に空ける配置だが、それでも同居の取り合いはあるため「方向性」の数字。）

同僚に却下されないよう、条件を揃えた。

- **被験体**: Rust = 既存 `lastshot`（axum + tokio + sqlx）。Node = 新規 `lastshot-node`（Fastify + postgres.js）。
  **同一 API・同一 SQL・同一レスポンス JSON**（片方を変えたら両方直す。パリティが崩れると比較が無意味）。
- **同一 Postgres**（`fsync=off` 等チューニング済み）・**同一クエリ**・**同一接続 = unix ソケット `/tmp`**。
- **負荷ツール**: [oha](https://github.com/hatoo/oha)（同一ツールで両方を叩く）。各セルで warmup 後に計測（既定 10s）。
- **本番ビルド**: Rust `cargo build --release`（opt-level3 / LLVM、profile 上書きなし） /
  Node `NODE_ENV=production` ＋ JIT ウォームアップ。
- **DB 接続予算を固定**: total pool（単一 = `POOL_MAX`、cluster は W worker で各 `POOL_MAX/W` にして総和一致）。
- **並列 2 モードを両方測る**:
  - **単一プロセス（1コア対1コア）** = `TOKIO_WORKER_THREADS=1` vs Node 1プロセス。**最もクリーン**。
  - **全コア** = 8 worker / 8 thread。コア数を揃える（macOS は taskset が無いので worker 数で揃える）。
- **メトリクス**: req/s ・ p99 ・ **CPU秒/1k req**（「同じ速度でもコストは別」の決め手）。

### エンドポイント（DB の重さを軸にスライド）

| パス | 中身 | 何を示すか |
| --- | --- | --- |
| `GET /ping` | DB なし `{"ok":true}` | ランタイム+HTTP+JSON の**素の天井**。DB を言い訳にできない領域。 |
| `GET /db/light` | `select value from counter where id=1`（点取得1往復） | **実 API の大半が住む現実ケース**。 |
| `GET /db/light_pipe` | 上と同一 SQL・同一レスポンス。Rust のみ、tokio-postgres の**パイプライン**版 | 多コア点SELECT の差がドライバ由来であることの検証。 |
| `GET /db/heavy` | `bench_rows`(30万行) を `like '%abc%'` 全走査して集約 | **主張が正しくなる唯一の領域**（収束する境界）。 |
| `GET /db/sleep?ms=N` | `pg_sleep(N/1000)`（PG CPU ほぼ0の純待ち） | ロック/IO 待ちで遅い場合＝両者収束を正直に提示。 |

### 既知の制約

- 負荷生成(oha)が同一マシン同居。全コアモードは oha+PG とコアを奪い合うため「方向性」の数字。
  **最もクリーンなのは単一プロセス（1コア対1コア）**。
- CPU は `ps %cpu` サンプルの平均なので近似値。
- PG の `max_connections` は既定 100 だと sqlx プール + パイプラインプールが同居したとき足りない
  （後述）。計測機は `ALTER SYSTEM` で 500 に引き上げた。

## 結果① 単一プロセス（1コア対1コア）＝最もフェア → Rust 圧勝

### /ping（DB なし）

| 同時数 | Rust req/s | Node req/s | Rust優位 | Rust p99 | Node p99 |
|---:|---:|---:|---:|---:|---:|
| 8 | 183,846 | 99,502 | 1.85× | 0.08ms | 0.15ms |
| 256 | 194,374 | 99,029 | 1.96× | 1.88ms | 3.15ms |

→ DB を一切使わずに約2倍。CPU も Rust が半分。**「DB のせい」が通らない領域で差が出る。**

### /db/light（点SELECT＝実 API の大半）

| 同時数 | Rust req/s | Node req/s | Rust優位 | Rust p99 | Node p99 |
|---:|---:|---:|---:|---:|---:|
| 1 | 19,997 | 14,754 | 1.36× | 0.07ms | 0.09ms |
| 32 | 64,332 | 27,170 | 2.37× | 0.67ms | 1.88ms |
| 128 | 63,707 | 14,614 | 4.36× | 2.62ms | 13.92ms |
| 256 | 62,547 | 11,821 | **5.29×** | 5.31ms | **33.01ms** |

→ **現実ケースで決定打。ただし倍率の読み方に注意。** 同時数 128→256 で **Rust はほぼ横ばい（63,707→62,547）なのに、
Node はむしろ低下（14,614→11,821）**している。つまり **5.29× の主因は「Rust が伸びた」ではなく「高負荷で Node が
性能を維持できず落ちた」こと**。p99 が c256 で **33ms** まで悪化しているのが、その実態を裏付ける。

正直な留保: これは「Rust が速い」というより「**単一プロセス × 高同時で Node 側（postgres.js / Fastify / イベントループ）が
詰まった**」とも読める。事実④で多コアにすれば Node は ~120k 出せるので、ここで見えているのは throughput の地力差というより
**1 プロセスのスケール特性の差**（イベントループ飽和・コネクション待ち行列）── これはこれで別の良い発見でもある。
それでも、CPU を 1req あたり約 5 倍食い、テール(p99)が桁で悪化する点は、Rust 有利の事実として残る。

### /db/heavy（重い集約＝DB 律速）

| 同時数 | Rust req/s | Node req/s | Rust優位 | Rust CPU秒/1k | Node CPU秒/1k |
|---:|---:|---:|---:|---:|---:|
| 256 | 275 | 262 | 1.05×（収束） | **約0.08** | 約0.28 |

→ スループットは収束（＝同僚の主張が成り立つ唯一の領域）。**だが Rust は CPU 約 1/3（≈0.08 vs ≈0.28 CPU秒/1k ＝ 約 3.4×）・
テール安定。「同じ速度」≠「同じコスト」。** ※この CPU 値は `ps %cpu` サンプル平均の近似なので、有効数字 1〜2 桁で読む
（3 桁目まで書くと精度を超えて見える）。

## 結果③ 追試: pool を 16→64 に上げたら Rust は抜き返すか？ → 抜き返せなかった（予想外れ）

multi-light のスループット（req/s）:

| 同時数 | Rust pool16 | Rust pool64 | Node pool16 | Node pool64 | pool64でのRust優位 |
|---:|---:|---:|---:|---:|---:|
| 32 | 65,974 | 70,089 | 92,271 | 108,101 | 0.65× |
| 64 | 66,349 | 76,999 | 119,565 | 113,432 | 0.68× |
| 128 | 66,827 | 77,072 | 97,842 | 116,889 | 0.66× |
| 256 | 66,939 | 76,723 | 76,668 | 121,314 | 0.63× |

- Rust は ~66k → ~77k（**+17%程度**）で頭打ち。Node(~120k)には届かず。
- **結論の訂正**: 「pool を増やせば Rust が追い抜く」は誤りだった。往復数を増やすだけでは不足で、
  Node に並ぶには pool 増ではなく **tokio-postgres 等で実際にパイプライン化**する必要がある。
  → この「多コア × 点SELECT」領域は **postgres.js（Node）が速い**のが実測。ただしこれは
  「言語の地力」ではなく「**ドライバ設計**」の差。

## 結果④ 決着: Rust もパイプライン化したら？ → Node を抜き返した

`/db/light_pipe`（Rust, tokio-postgres のパイプライン版。sqlx と同一 SQL/レスポンス）を追加し、
multi / pool=64 / `max_connections=500` で再計測:

| 同時数 | Rust sqlx | Rust pipe | Node | pipe/sqlx | pipe/Node |
|---:|---:|---:|---:|---:|---:|
| 32 | 70,441 | 121,415 | 107,354 | 1.72× | 1.13× |
| 64 | 76,867 | 128,666 | 112,547 | 1.67× | 1.14× |
| 128 | 76,921 | 132,899 | 117,035 | 1.73× | 1.14× |
| 256 | 76,826 | **134,744** | 121,175 | 1.75× | 1.11× |

p99（ms, c256）: Rust pipe **2.44** < Node 5.27 < Rust sqlx 3.83。

→ **「多コア × 点SELECT で Node が速い」は言語差ではなくドライバのパイプライン差**だったと確定。
Rust に同じ手札（パイプライン）を持たせると、スループットも p99 も Rust が勝つ。
これで全領域（1コア・/ping・heavy 効率・多コア点SELECT）で Rust 優位に揃った。

## パイプラインとは / そのデメリット

### パイプラインとは

DB に「1問ずつ聞いて答えを待つ」のではなく「**質問を連射してから答えをまとめて回収**」する方式。
クエリ1個ごとの往復待ち（round-trip）を重ねて消すので、「DB 処理は軽いが往復で詰まる」場面で効く。
postgres.js は既定でこれをやる。sqlx はやらない（1接続 = 1クエリ）。Rust でやるなら tokio-postgres
（futures を並行に poll すると自動でパイプラインされる。ただしサーバ側の実行は接続内で直列）。

### デメリット（速くなる代わりに何を払うか）

- **効くのは往復律速のときだけ。** クエリ自体が重い（CPU/IO 律速）と、往復はもう支配項ではないので
  ほぼ無風。実際 `/db/heavy` ではパイプラインの利得は出ない。
- **サーバ側は接続内で直列実行。** PG はワイヤプロトコル上、受け取った順に 1 接続内で逐次処理する。
  つまりパイプラインの利得は純粋に **往復(RTT)の隠蔽**であって、**1 接続のサーバ処理並列度は 1 のまま**。
  詰め込みすぎると後続クエリのレイテンシは積んだぶん延びる（HOL ブロッキング）。
  → 処理の**並列度**が欲しければ**接続を増やすしかなく、パイプラインと接続プールは別軸の最適化**である。
  これが③で「pool を増やしても届かず、pipe 化（往復隠蔽）が別途必要だった」理由そのもの。
- **エラー時の巻き込み・トランザクション境界が複雑化。** PostgreSQL のパイプラインは、途中で 1 つエラーが起きると
  **同一パイプライン内の後続クエリが暗黙に破棄され、次の `Sync` まで abort される**（implicit transaction 的な振る舞い）。
  「扱いが難しい」のではなく仕様としてこう動くので、明示トランザクションや「順序依存のある書き込み」とは相性が悪い。
- **メモリと公平性。** in-flight なクエリ/結果をバッファするぶんメモリを使う。1接続に偏ると
  接続間の負荷が偏り、レイテンシのばらつきが増えうる。
- **デバッグしづらい。** 「どのクエリがどの往復で詰まったか」が1往復1クエリより追いにくい。

要するにパイプラインは **「軽いクエリ × 高同時数」専用の強力な最適化**で、重いクエリや厳密な
トランザクションでは利得が無い／むしろ複雑さが増す。今回 Node が先行して見えたのは、この最適化が
postgres.js では既定 ON だったから（＝言語差ではなくドライバ設計差）。

### 補足: 今回の数字はパイプライン効果を「過小評価」している（unix ソケットゆえ）

今回の接続は `/tmp` の **unix ソケット**なので、物理的なネットワーク往復(RTT)は極小。それでもパイプライン化で
**1.7×** 出たということは、利得の正体は「ネットワーク往復の隠蔽」よりむしろ **カーネルの送受信 syscall 回数の削減**と
**tokio のタスクスケジューリング往復の削減**である可能性が高い。

含意: 本番のように **DB が別ホスト(TCP)** だと RTT が上乗せされるぶん、パイプラインの利得は **今回よりさらに大きくなる**はず。
つまり **今回の数字はパイプライン効果を過小評価している側**であり、「ローカルソケットでこれなら、本番（TCP）ではもっと差が開く」
と読める（＝レポートの射程はむしろ広がる方向）。

### 計測上の注意（接続予算）

sqlx プール(64) と pipe プール(64) が同居すると合算 128 本になり、PG の `max_connections`（既定100）を
超えて `too many clients` で全滅する。計測機は 500 に引き上げて回避した。

## 同僚の主張への評価（まとめ）

| 主張 | 実測 |
|---|---|
| 「速度は変わらない」 | **ほぼ全領域で崩れる**（差は出る）。同じになるのは重DBのスループットだけ。 |

→ 「DB 遅いから一緒」は最重量クエリのスループットだけを見た一般化で、軸を振ると崩れる。
ただし「Rust なら何でも速い」も誇張で、本質は **スタック/ドライバ/コア配分**で決まる。

## 再現方法

詳細は実装ブランチ [`bench-rust-vs-node`](https://github.com/daijinload/eikodan/tree/bench-rust-vs-node)
の `lastshot-node/README.md` 参照。要点のみ:

```sh
# 0) ネイティブ Postgres 起動 + bench_rows migration 適用（lastshot 側で1回）
cd lastshot && ./run db-start && ./run db-setup && ./run db-migrate

# 1) Node 依存
cd ../lastshot-node && npm install

# 2) フルベンチ（Rust release ビルド → 計測 → REPORT.md 生成まで一気通し）
cd bench && ./bench.sh

# 短縮実行（軸を絞る）
DURATION=5 CONNS="1 64 256" ENDPOINTS="ping light heavy" ./bench.sh
```
