# コンテナ運用メモ（CI / ローカル）

GitHub Actions でフルテストを回すにあたり、コンテナ構成をどうするかを整理したメモ。
「全部入りシングルコンテナ vs compose 分割」「1コンテナ1プロセス原則」「停止のやり方」あたりの結論を残す。

## 結論（最初にこれだけ）

1. **「同じコンテナだから爆速」は誤解。** 速さの正体は Postgres のデータを tmpfs（RAM）に置いて
   `fsync=off` 等にしていること。コンテナ構成（同居か分割か）とは無関係で、どっちでも速くできる。
2. **「1コンテナ1プロセス」は本番（k8s等）の都合。** 使い捨ての CI / dev には当てはまらない。
   全部入りシングルコンテナはむしろ「ローカルと CI を同一イメージにして"CIだけコケる"を潰す」用途に刺さる。
3. **複数プロセスの唯一の実コストは PID1 のシグナル/ゾンビ処理。** ただし `exec` か `--init` の
   どちらかを知っていれば1行で回避できる。
4. **停止は `down -t 0`（即SIGKILL）で統一。** 使い捨て前提なら綺麗なシャットダウンが要らないので、
   これでローカルも CI も問題ない。上の3の考慮すら実質不要になる。

## 1. 「同じコンテナなら速い」の誤解

「Postgres を同じコンテナに入れると爆速」と言われがちだが、効いている理由がズレている。

- **同居の通信ゲインはほぼ無い** — 別コンテナでも同一ホスト上ならどちらも loopback 通信。
  アプリ⇔Postgres のネットワーク差は誤差レベル。
- **本当に効くのはここ** — データディレクトリを tmpfs（メモリ）に置き、
  `fsync=off` / `full_page_writes=off` / `synchronous_commit=off` にする。
  ディスク I/O と耐久性保証を捨てるから速い。CI は使い捨てなのでこれが許される。

この設定は「シングルコンテナ」でも「サービスコンテナ」でも「ランナー直インストール」でも同じように効く。
**速さとコンテナ構成は独立した話。**

## 2. GitHub Actions での構成の選択肢

| 方式 | 速さ | 手間 | 備考 |
|---|---|---|---|
| `services:` で Postgres をサイドカー（tmpfs指定） | ◎ | 小（YAMLだけ） | GHA の定石 |
| ランナーに Postgres 直起動（tmpfs） | ◎ | 小 | |
| 全部入りシングルカスタムイメージ | ◎ | 大（イメージ保守＋プロセス管理） | ローカル/CI 完全同一が狙えるのが利点 |

GHA のランナー自体がもう使い捨て VM なので、速さ目的だけなら全部を1コンテナに詰める旨味は薄い。
**シングルコンテナは「速さ」ではなく「ローカルと CI で完全に同じ環境を再現したい（CIだけコケるを無くす）」
ときに選ぶもの。** その代わりプロセス管理とイメージ保守のコストが乗る。

サービスコンテナを tmpfs で動かす例:

```yaml
services:
  postgres:
    image: postgres:16
    env:
      POSTGRES_PASSWORD: postgres
    options: >-
      --mount type=tmpfs,destination=/var/lib/postgresql/data
      --health-cmd pg_isready --health-interval 5s
    # command 相当で -c fsync=off -c full_page_writes=off を足すとさらに削れる
```

## 3. 「1コンテナ1プロセス」原則はどこまで本気か

結論：**本番でオーケストレータ（k8s 等）に載せる時の都合**であって、CI / dev には当てはまらない。

- **1プロセス原則が要るのは本番だけ** — プロセス単位で独立スケール・独立再起動・ログ分離・
  オーケストレータのヘルスチェックをやりたいから。CI は使い捨ての1ユニットなので、その要件が全部消える。
- **「全部入り」が刺さるのはむしろ CI / dev** — ローカルと CI で完全に同じ1イメージを動かせる＝
  「CIだけコケる」の温床を潰せる。これは環境差をなくしたい時の本命。

## 4. 複数プロセスを1コンテナで回す時の実コスト

「面倒」なのは起動順ではなく、**コンテナを止める時**と**死んだ子プロセスの後始末**。

### (1) `docker stop` / `down` がきれいに効かない
コンテナ内で最初に動いたプロセスが **PID 1** になり、`docker stop` の SIGTERM を子（postgres/アプリ）に
転送する責務を負う。起動スクリプトをこう書くとシェルが PID1 になり、シェルは SIGTERM をうまく転送しない:

```sh
postgres &
./app          # ← シェル経由だとシェルが PID1。SIGTERM が postgres/app に届かない
```

結果 → `docker stop`（や `down`）は SIGTERM 送信後 **既定10秒待って** ダメなら SIGKILL。
**毎回10秒待ち**になる。

### (2) ゾンビプロセスが溜まる
PID1 には「親が先に死んだ孤児プロセスを引き取って後始末（reap）する」義務もある。
普通のシェルやアプリはこれをやらないので、長時間動かすと `<defunct>` がじわじわ溜まる。
**これはコンテナが動いている最中の問題**なので、停止コマンドでは解決しない。

### (3) 「準備完了待ち」が要る
起動順を守っても、postgres が「起動した」のと「接続を受けられる」のは別。
`pg_isready` で待たないと、アプリが先に繋ぎに行って初回だけコケる。

### 回避は1行
- **CI など使い捨て前提** → 起動スクリプトの最後を `exec ./app` にする
  （シェルを残さずアプリを PID1 に置き換える）。これで (1) はほぼ消える。
- **ローカルでも使い回す** → エントリポイントに `tini` を噛ませる。
  Docker なら `docker run --init` 一発でゾンビ回収とシグナル転送を全部やってくれる。

## 5. 停止は `down -t 0` で統一

`docker compose down`（猶予つき）と `down -t 0`（即殴り殺し）の違いに注意:

- **`docker compose down`** → SIGTERM → 既定10秒待ち → SIGKILL。
  PID1 がシグナル転送できていないと、**この10秒をフルで待つ**。downは問題を回避せず、むしろ踏むコマンド。
- **`docker compose down -t 0`** → 猶予ゼロで即 SIGKILL。待ち時間ゼロ。
  4節の (1)(2) を**丸ごと回避**できる（綺麗に止める必要が無くなるので `exec`/`--init` も実質不要）。

**本プロジェクトの方針：ローカルも CI も停止は `docker compose down -t 0` で統一。**
使い捨て・毎回作り直し前提なので綺麗なシャットダウンは要らず、これで足りる。

唯一の留意点：SIGKILL なのでコンテナ内データは綺麗に閉じられない。
将来「長生きさせてデータを再起動後も残したいコンテナ」が出てきた時だけ `-t 0` は不利になる。
今の使い捨て運用では無関係。

## 6. 計測予定（保留中）

「compose 分割」と「全部入りシングルコンテナ」の比較計測は、リポジトリに**最終形態のアプリが
出来てから**やる。論点は速さ（これは tmpfs/fsync 次第でどちらも出る）ではなく、
**起動時間・CI総時間・ローカル再現性**が compose 分割に対して割に合うか。
