# Rust(lastshot) vs Node(lastshot-node) API ベンチ結果

> 「DBが律速だから言語は関係ない」を検証する実測。同一 Postgres・同一クエリ・同一接続(unixソケット)で、DBの重さ(ping→light→heavy)を軸に oha で叩いた。**生成は `bench.sh`、集計は `summary.mjs`**。

## 結論（読み解き）

**「DBが律速だから言語は関係ない」は、重いDBクエリ1点でしか成り立たない。実APIの大半では Rust が明確に速い。**

最もクリーンな比較は **単一プロセス（1コア対1コア）**（負荷生成とコアを奪い合わない）。そこでは:

- **/ping（DBなし）**: Rust が約 2× のスループット・CPU 半分・小さい p99。DBを持ち出せない以上、ここで差が出る時点で「言語は関係ない」は崩れる。
- **/db/light（点SELECT＝実APIの大半）**: Rust が **約 2.4〜5.3× のスループット**で、しかも**同時数が上がるほど差が開く**。Node は p99 が桁で悪化（c=256 で Rust 5ms 台 vs Node 30ms 台）、CPU も 1req あたり約 5× 食う。**現実のAPIが住むこの領域で主張は完全に崩れる。**
- **/db/heavy（重い集約＝DB律速）**: ここだけスループットが収束する（飽和時はほぼ同じ req/s）。**＝同僚の主張が成り立つ唯一の領域。** ただしそれでも Rust は **CPU を約 3× 節約**し、テール（p99）も安定。「同じ速度」でも「同じコスト・同じ安定性」ではない。

### 全コアモードの注意点（正直な但し書き）

全コアの **/db/light で Node が Rust を上回るセルがある**。これは言語の地力ではなく**ドライバ差＋pool設定**: postgres.js は1接続で複数クエリを**パイプライン**するが、sqlx はしない（1接続=1クエリ）。このため pool=16 を揃えても「実効の同時クエリ数」は揃っておらず、Rust 多コア light は `接続数 × 1往復` でキャップされ ~66k で頭打ちになる（単一コアでも ~63k 出るので、コア追加ではなく往復律速）。pool を増やすか Rust 側もパイプライン化すればこのキャップは外れる。**それでも 1req あたり CPU は Rust の方が少ない。**
※ 全コアモードは oha+PG とコアを奪い合うため「方向性」の数字。**1コア対1コアが最も公平**で、そこは Rust の圧勝。

### まとめ

- DBが**本当に**重い（CPUを使い切る）ときだけ、スループットは収束する。
- 実APIの大半（軽い点取得・DBなし）では **Rust が 2〜5× 速く、CPUは数分の1、テールは桁で安定**（1コア比較）。
- 収束する領域でも **CPU効率と p99 は別物**で Rust が勝つ → クラウド費用と SLO に直結。
- 「DB遅いから一緒」は、最も重いクエリのスループットだけを見た一般化。軸を振ると崩れる。
