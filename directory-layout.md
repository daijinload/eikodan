# ディレクトリ構成方針（設定ファイルの置き場所）

eikodan リポジトリで「設定ファイルをルート直下に置くか、フォルダにまとめるか」を
判断するときの方針メモ。

## 結論

「ルート直下が正しい」のではなく、**多くのツールが設定を“親方向に自動探索”する都合で
そこに置くしかない**ものが多い、というのが実態。だから「formatter もブラウザテストも
全部フォルダにまとめる」を全面適用すると、自動探索とエディタ連携（保存時フォーマット・
rust-analyzer 等）が効かなくなり、各所で `--config パス` を手配線する羽目になって
かえって面倒になりやすい。

ただし「フォルダにまとめたい」という感覚は正しい。対立して見えるのは実は
**「モノレポのルートに“1個の共有設定”を置く」案 vs「フォルダごと」案** の対立であって、
“フォルダにまとめること”自体とは衝突しない。

## カギになる事実

設定の自動探索は、ツールを**起動した場所**を基準に親ディレクトリをたどる。だから—

- ❌ ルートに共有 `playwright.config.ts`、`tests/` だけ別フォルダ → 中途半端で配線が要る
- ✅ `e2e/` を丸ごと自己完結させ、その中に config もテストも置き、ツールはその
  フォルダから起動 → 「フォルダにまとまっている」かつ「自動探索も効く」の両取り

## 方針（ルール）

1. **自動探索アンカー系の設定**
   （`compose.yaml` / `Cargo.toml` / `rust-toolchain.toml` / `rustfmt.toml` /
   `.sqlfluff` / `playwright.config.ts` など）
   → その**関心事フォルダ／サブプロジェクトのルート**に置き、ツールはそのフォルダから動かす。
   ツールと戦わない。

2. **関心事の“中身”**（スクリプト・カスタムルール・fixture・semgrep など）
   → 関心事フォルダにまとめる。

3. **共有したいとき**は、ツールが対応している場合のみ
   「ルートに薄い参照ファイル＋本体はフォルダ」にする
   （compose の `include:`、tsconfig/eslint の `extends` 等）。
   `compose` / `rustfmt` は extends が弱いので無理に共有しない。

要するに **「関心事ごとの自己完結フォルダ」を単位にする** こと。
これで「フォルダにまとめたい」を満たしつつ、自動探索とも喧嘩しない。

## リポジトリ内の手本

| 例 | やっていること |
|----|----------------|
| `playwright-sample/` | 関心事を1フォルダに自己完結（config がそのフォルダのルート）→ 理想形 |
| `fastweb/assets/` | CSS/Tailwind の“実装”（スクリプト＋semgrep ルール）を1フォルダに集約。discovery 対象の `Cargo.toml` 等はプロジェクトルート |
| `task-runners/` | ツール種別ごとにフォルダ分け（showcase なのでこれが最適） |
